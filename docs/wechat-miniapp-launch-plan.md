# CarbCycle Tracker 上线与微信小程序迭代计划

> 适用范围：
> 1. 先把当前 Next.js / PWA 版本做成可正式上线的 Web 产品
> 2. 基于现有 Web 版本快速接入微信小程序 `web-view`
> 3. 如果后续数据验证和用户规模成立，再评估迁移到原生微信小程序

---

## 1. 目标与建议路线

### 1.1 当前项目现状

当前项目已经具备：

- 完整的移动端页面框架
- 饮食、计划、统计、记录、设置等主要页面
- PWA 基础能力
- Prisma 数据模型
- 基础 API 校验与限流
- Docker / Vercel / GitHub Actions 基础部署文件

但目前仍然更接近 **可演示的产品雏形**，还不是完全可长期运营的正式产品。

### 1.2 当前最合理的上线顺序

建议按下面的路线推进：

1. 先把 **Web / PWA 正式版** 做到可用
2. 再做 **微信小程序 `web-view` 壳版本**
3. 最后再决定是否投入做 **原生微信小程序**

### 1.3 为什么这么做

原因：

- 当前代码是 Next.js Web 应用，不能直接变成微信小程序原生项目
- 先上线 H5 / PWA 可以最快验证需求和留存
- `web-view` 小程序成本最低，适合快速分发
- 原生小程序要重做前端承载层，工期明显更长

## 2. 路线总览

### 路线 A：Web / PWA 正式上线

目标：

- 让当前项目成为可长期运行、可多用户使用、可稳定部署的正式 Web 产品

预计周期：

- `1 ~ 2 周`

### 路线 B：微信小程序 `web-view` 版本

目标：

- 使用小程序外壳承载当前 H5 页面，在微信生态内快速可访问

预计周期：

- `1 ~ 3 周`

### 路线 C：原生微信小程序版本

目标：

- 使用 `Taro` 或其他多端框架重构前端，做成真正的原生小程序体验

预计周期：

- `3 ~ 6 周`

## 3. 当前必须补齐的问题

以下问题建议优先解决，否则即使能上线，也不适合正式对外运营。

### 3.1 用户数据仍然以本地状态为主

当前引导和大量业务状态主要存在前端 Zustand 中，本地可用，但不是真正的账号数据体系。

需要补齐：

- 用户注册 / 登录 / 会话机制
- 用户资料持久化到数据库
- 用户计划、餐食、身体记录、训练记录与饮水记录入库
- 多设备同步

### 3.2 存在 demo 回退数据

当前部分页面在数据库失败时会回退到 demo 数据。

需要处理：

- 区分演示模式与正式模式
- 正式环境下禁止静默回退 demo 数据
- 接口失败时给用户明确错误提示

### 3.3 API 已有基础校验，但还不够企业级

当前已加：

- Zod 校验
- 基础限流

还建议补齐：

- 登录鉴权
- 用户级权限校验
- 操作审计日志
- 错误监控与报警
- 更可靠的分布式限流（Redis / Upstash）

### 3.4 小程序能力与 Web 能力存在差异

当前这些能力在原生小程序上不能直接照搬：

- Service Worker / PWA 安装提示
- `BarcodeDetector`
- `Blob` / `URL.createObjectURL`
- Next.js 路由
- 一部分 Web 图表和 DOM 交互

## 6. Phase 2：微信小程序 `web-view` 快速上线方案

目标：

- 最低成本接入微信入口，让用户可从微信内访问当前产品

### 6.1 小程序方案定义

该方案不是把当前项目直接编译成小程序，而是：

- 新建一个微信小程序项目
- 首页或主页面使用 `web-view`
- `web-view` 指向你线上 H5 地址

### 6.2 需要完成的工作

#### 小程序后台

- [ ] 注册并认证微信小程序
- [ ] 配置业务域名
- [ ] 配置请求合法域名
- [ ] 配置上传域名 / 下载域名（如果有）
- [ ] 配置隐私政策与个人信息说明

#### 小程序前端壳

- [ ] 新建小程序项目
- [ ] 创建启动页
- [ ] 创建 `web-view` 页面
- [ ] 添加基础导航与 loading
- [ ] 增加登录态桥接方案

#### H5 配合

- [ ] 上线正式 HTTPS 域名
- [ ] 处理从微信内打开的兼容样式
- [ ] 准备分享落地页 / 跳转页
- [ ] 处理微信环境下的登录与授权

### 6.3 重点风险

- `web-view` 体验仍然本质上是 H5
- Web 的 PWA 安装能力在微信环境下不重要
- 扫码、支付、分享等能力如果要深度接微信，后面仍可能要做原生

### 6.4 适合采用这条路线的条件

- 你想先验证产品是否有人用
- 你需要尽快进入微信生态
- 你当前团队人力有限

### 6.5 Phase 2 验收标准

- 小程序可通过审核
- 用户在微信内能稳定打开 H5
- 登录、查看、记录三个核心路径可用

## 7. Phase 3：原生微信小程序重构方案

目标：

- 获得更好的微信生态体验和更强的原生能力支持

### 7.1 技术路线建议

推荐：

- `Taro + React + TypeScript`

原因：

- 你当前已经是 React + TS
- 逻辑层可复用度更高
- 更适合后续同时维护 H5 和小程序

### 7.2 可复用部分

以下内容建议抽成共享业务包：

- [ ] 营养与计划计算逻辑
- [ ] 领域模型类型
- [ ] Zod 校验逻辑
- [ ] API 请求层
- [ ] 文案常量 / 枚举映射

当前可直接参考的目录：

- [`lib/calculator.ts`](../lib/calculator.ts)
- [`lib/domain.ts`](../lib/domain.ts)
- [`lib/validation.ts`](../lib/validation.ts)
- [`lib/ui-config.ts`](../lib/ui-config.ts)

### 7.3 必须重写部分

- [ ] 路由与页面容器
- [ ] 所有 UI 组件
- [ ] 图表层
- [ ] 扫码组件
- [ ] 上传组件
- [ ] 本地缓存与离线策略
- [ ] 权限申请流程

### 7.4 Web API 到小程序 API 映射

#### 当前 Web 做法

- `BarcodeDetector`
- `fetch`
- `localStorage / persist`
- 浏览器文件上传

#### 小程序替代

- `wx.scanCode`
- `wx.request`
- `wx.setStorageSync / wx.getStorageSync`
- `wx.chooseMedia / wx.uploadFile`

### 7.5 页面迁移顺序

建议按价值顺序迁移：

1. [ ] 首页 Dashboard
2. [ ] 添加食物
3. [ ] 引导页
4. [ ] 身体记录
5. [ ] 计划页
6. [ ] 统计页
7. [ ] 设置页

### 7.6 原生小程序版额外可做能力

- [ ] 微信登录
- [ ] 原生扫码
- [ ] 原生分享
- [ ] 模板消息 / 订阅消息
- [ ] 原生上传图片
- [ ] 更轻量的列表性能优化

### 7.7 Phase 3 验收标准

- 不依赖 `web-view`
- 原生完成主要记录流程
- 扫码、上传、登录都使用小程序原生能力

## 11. 建议你现在直接开始做的事情

如果你想按最短路径推进，我建议马上按下面顺序开始：

1. 先做 **用户登录 + 数据持久化**
2. 再做 **去 demo fallback**
3. 然后做 **生产环境发布**
4. 发布后做 **小程序 `web-view` 壳**
5. 最后根据数据决定是否进入原生小程序重构
