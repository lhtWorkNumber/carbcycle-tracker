# CarbCycle Tracker 执行顺序清单

> 目标：
> 按最小阻塞路径推进，把当前项目从“可演示 demo”逐步推进到“可上线 Web 版”，再进入微信小程序阶段。

---

## 执行原则

### 原则 1：先打通真实数据，再做平台扩展

先把 Web 版做成真正可用产品，再考虑微信小程序。

### 原则 2：按依赖顺序做

不要先做后面页面优化，再回头补用户体系和数据库。

### 原则 3：每个任务都要有验收结果

做完一项，就能明确判断“是否完成”。

---

## Milestone 1：Web 正式版可上线

这一阶段是当前最优先目标。

---

## Task 1：确定用户体系与鉴权方案

### 目标

给项目增加正式用户登录，不再依赖本地单用户状态。

### 要做的事

- 选定鉴权方案
- 明确用户注册 / 登录 / 会话机制
- 明确前端如何拿当前用户
- 明确后端如何识别当前用户

### 建议方案

- 使用 `Supabase Auth`

### 完成标准

- 能清楚写出登录架构
- 环境变量与依赖准备完成
- 可以进入下一任务开发

---

## Task 2：接入登录与用户上下文

### 目标

前端和后端都具备“当前登录用户”的概念。

### 要做的事

- 接入登录页 / 登录入口
- 保存用户 session
- 封装当前用户上下文
- 增加未登录态处理

### 完成标准

- 用户可以登录
- 登录后前端能拿到用户身份
- 刷新页面不会立即丢 session

---

## Task 3：引导页资料正式入库

### 目标

当前引导页不能只写 Zustand，要写入数据库。

### 要做的事

- 引导完成时调用 `/api/users`
- 把引导结果落到 `User`
- 本地 store 改成以服务端数据为主

### 相关位置

- `components/screens/onboarding-flow.tsx`
- `app/api/users/route.ts`
- `store/tracker-store.ts`

### 完成标准

- 完成引导后数据库中有真实用户记录
- 刷新页面后资料仍存在

---

## Task 4：建立当前用户数据加载流程

### 目标

页面打开时应读取当前用户真实数据，而不是靠 demo 初始值。

### 要做的事

- 应用启动后读取当前用户资料
- 如果有服务端数据，覆盖本地默认值
- 处理加载中 / 未登录 / 读取失败状态

### 完成标准

- 页面首屏不再依赖 demo 用户
- 当前用户数据来自服务端

---

## Task 5：餐食记录正式写库

### 目标

添加食物页和首页餐次记录要接真实后端。

### 要做的事

- 添加食物后调用 `/api/meal-logs`
- 首页从真实餐食记录读取
- 模板应用后批量写入真实餐食

### 相关位置

- `components/screens/add-food-screen.tsx`
- `components/screens/dashboard-screen.tsx`
- `app/api/meal-logs/route.ts`

### 完成标准

- 添加食物后刷新页面仍能看到
- 不依赖本地 demo 餐食数据

---

## Task 6：身体记录正式写库

### 目标

身体记录页写入真实后端并回显。

### 要做的事

- 表单提交调用 `/api/body-records`
- 历史记录从后端加载
- 处理空状态与错误状态

### 完成标准

- 保存后刷新仍可见
- 新增记录出现在历史列表中

---

## Task 7：运动记录正式写库

### 目标

运动记录页写入真实后端并展示当天汇总。

### 要做的事

- 表单提交调用 `/api/exercise-logs`
- 今日汇总从真实数据计算
- 历史记录从后端加载

### 完成标准

- 保存后刷新仍存在
- 汇总数据正确变化

---

## Task 8：每日计划正式写库

### 目标

碳循环计划不只在本地推导，要支持真实保存与读取。

### 要做的事

- 周计划生成结果写入 `/api/daily-plans`
- 首页 / 计划页使用真实计划数据
- 手动编辑目标时同步后端

### 完成标准

- 计划可持久化
- 刷新后不丢失编辑内容

---

## Task 9：去掉正式路径中的 demo fallback

### 目标

正式环境下不能静默回退到 demo 数据。

### 要做的事

- 检查 `demo-data.ts` 的使用点
- 区分 demo 模式与正式模式
- 正式模式下失败要报错，不要假装成功

### 优先处理

- `app/(shell)/add-food/page.tsx`
- `app/(shell)/settings/page.tsx`
- `store/tracker-store.ts`

### 完成标准

- 正式路径不再隐式依赖 demo 数据

---

## Task 10：模板系统正式写库

### 目标

饮食模板能跨设备保存与复用。

### 要做的事

- 设计 `MealTemplate` / `MealTemplateItem` 模型
- 保存模板到数据库
- 模板列表从后端读取
- 应用模板时使用真实数据

### 完成标准

- 新建模板后刷新仍在
- 登录同一账号跨设备可见

---

## Task 11：饮水记录正式写库

### 目标

饮水追踪成为正式可累计的数据。

### 要做的事

- 设计 `WaterLog` 模型
- 快速加水量写库
- 今日目标读写后端

### 完成标准

- 加水后刷新仍保留
- 饮水目标可持久化

---

## Task 12：周报系统正式写库

### 目标

周报不是每次前端即时推导，而是可复用的历史报告。

### 要做的事

- 设计 `WeeklySummary` 模型
- 生成每周周报
- 保存本周与上周对比

### 完成标准

- 周报可历史查看
- 数据口径稳定

---

## Task 13：成就系统正式持久化

### 目标

成就状态与进度属于用户历史数据，不应只靠前端推导。

### 要做的事

- 设计 `AchievementProgress` 模型
- 把 streak / 打卡 / 减重进度持久化
- 设置页与首页读取真实成就状态

### 完成标准

- 成就不会因缓存丢失而重置

---

## Task 14：统一服务端聚合读取接口

### 目标

前端不应到处自己拼数据，应该有统一“当前用户首页数据”读取方式。

### 要做的事

- 增加 dashboard 聚合接口
- 增加 settings 聚合接口
- 减少前端在多个地方重复拼接状态

### 完成标准

- 首页和设置页有更稳定的数据来源
