# CarbCycle Tracker 迭代排期路线图

> 目标：
> 按最短可上线路径推进，优先完成 Web / PWA 正式版，再接微信小程序 `web-view`，最后评估原生小程序迁移。

---

## 总体节奏

建议按 6 周推进：

- 第 1 周：数据与鉴权基础
- 第 2 周：核心记录写库与真实数据替换
- 第 3 周：统计、模板、饮水、周报产品化
- 第 4 周：生产部署与稳定性
- 第 5 周：微信小程序 `web-view` 壳与提审准备
- 第 6 周：联调、修复、正式发布

---

## 第 1 周：用户与数据基础

### 目标

- 把“本地 demo”变成“有用户、有后端数据”的应用

### 本周任务

- [ ] 选定鉴权方案
- [ ] 接入用户登录
- [ ] 设计当前用户上下文
- [ ] 完成用户资料入库
- [ ] 引导页完成后写库，不再只写本地 store
- [ ] 统一前后端 `user_id` 逻辑

### 建议优先实现

- Supabase Auth
- `User` 数据与登录态联动

### 验收标准

- 用户能注册 / 登录
- 用户资料能持久化
- 刷新页面后数据仍可恢复

---

## 第 2 周：核心记录正式写库

### 目标

- 核心业务数据不再依赖 demo fallback

### 本周任务

- [ ] 餐食记录正式对接 `/api/meal-logs`
- [ ] 身体记录正式对接 `/api/body-records`
- [ ] 运动记录正式对接 `/api/exercise-logs`
- [ ] 每日计划正式对接 `/api/daily-plans`
- [ ] 设置页读取真实用户资料
- [ ] 去掉正式路径上的 demo fallback

### 建议改造顺序

1. 首页读取真实餐食数据
2. 添加食物页写入真实数据
3. 身体记录页写入真实数据
4. 运动页写入真实数据

### 验收标准

- 添加 / 修改记录后刷新页面数据不丢失
- 不依赖 Zustand 本地初始化数据也能工作
- 核心页面无 demo 回退

---

## 第 3 周：智能功能产品化

### 目标

- 智能推荐、饮食模板、饮水、成就、周报从“前端推导”变成“真实产品能力”

### 本周任务

- [ ] 为饮食模板新增数据库模型
- [ ] 为饮水记录新增数据库模型
- [ ] 为周报新增数据库模型
- [ ] 为成就状态新增数据库模型
- [ ] 智能推荐改为基于真实数据库与用户实际记录
- [ ] AI 饮食建议改为服务端生成逻辑

### 建议新增表

- `MealTemplate`
- `MealTemplateItem`
- `WaterLog`
- `WeeklySummary`
- `AchievementProgress`

### 验收标准

- 模板保存后跨设备可见
- 饮水记录刷新后不丢失
- 周报可复用历史数据
- 成就状态可持续累计

---

## 第 4 周：生产部署与稳定性

### 目标

- Web / PWA 达到可正式发布标准

### 本周任务

- [ ] 切到 Supabase PostgreSQL 生产环境
- [ ] 跑通 Prisma 生产迁移
- [ ] 接入 Sentry 或等价错误监控
- [ ] 完成 Vercel 部署
- [ ] 验证 Docker 部署
- [ ] 增加 staging 环境
- [ ] 完成 PWA 离线缓存验证
- [ ] 修复 `build -> dev` 切换导致的 `.next` 污染问题

### 验收标准

- 生产环境可部署
- 生产数据库迁移可执行
- 错误可观测
- PWA 安装与离线回退可用

---

- [x] 创建 `web-view` 页面

### 目标

### 本周任务

---

## 上线后第一个迭代周期

上线后一周建议重点观察：

- 登录转化率
- 引导完成率
- 日活 / 次留 / 七留
- 餐食记录完成率
- 周报查看率
- 小程序打开成功率

建议根据这些数据决定是否进入原生小程序迁移。

---

## 原生小程序何时开始

只有满足以下条件，才建议开始：

- Web 版留存稳定
- 小程序 `web-view` 有稳定访问量
- 用户对扫码、分享、登录体验有明显诉求
- 你愿意投入额外 3~6 周重构前端
